# 版本升级前「旧版本消息已全部落块」检测

## 为什么要这一步（即便难度看似不变）

新版本若改了出块难度（`nmsci.register/transaction-difficulty-target-nbits`），而升级时旧版本还有**未落块的积压消息**被新版本带走打包，可能使链**永久验不过**：

- 每条 PoW 消息自带难度字段（中心签名覆盖、不可变）；区块头难度按出块时配置写入。
- 验证器要求：每条 PoW 消息的难度 == 前一个区块的难度（见 `ChainVerifier.difficultyMatchesIngestion`）。
- 旧难度的积压消息若**跨越不止一个新版本区块**，从第二个新块起就会与「前一个新版本块」的新难度不一致 → 整链不可逆地验不过。

因此每次升级前，统一确保旧版本已受理的消息**全部落进旧版本区块**、新版本从空池起步——**与难度是否变化无关**，作为升级的统一前置。

> 版本轴、源码哈希轴没有「块内消息 vs 本块头」的耦合检查，只有难度轴是 tip 相对；但本前置对所有升级统一执行最稳妥。

## 流程

1. **冻结入口（运维侧）**：在网络层停止对外暴露消息提交端口（关闭/隔离端口，或在反向代理层对写入请求返回 503），使不再有新消息进入。应用本身不内置冻结开关。
2. **排空**：让旧版本继续出块，把积压打包进旧版本区块（重启会触发首跑全量落块，或等出块周期 tick）。
3. **检测**：确认待落块消息为 0：

   ```sql
   select count(*) from msg_abstracts where is_in_block = false;  -- 必须为 0
   ```

   为排除竞态，间隔数秒复查一次，并确认期间 `msg_abstracts` 总数不再增长（证明入口确已停止受理）。为 0 且稳定后方可升级。

> 难度/版本对齐等其它上线前置见 docs/DB-BASELINE.md。
